# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`pinnae` (working name: `squeaker-listen`) — a real-time rat USV (ultrasonic vocalization) detector. It listens via a Dodotronic Ultramic SO.104 at 192 kHz, detects calls in defined frequency bands using FFT thresholding, and logs events to SQLite. It is a companion tool to Squeakorithm (separate repo), which plays enrichment audio; the two communicate only through a thin session manifest JSON.

See `ARCHITECTURE-PLAN.md` for the full rationale, phased roadmap, and band/overlap analysis.

## Setup and running

```bash
pip install -r requirements.txt

python listen.py devices                                        # find the SO.104 device name
python listen.py baseline --device "Ultramic SO.104" --db events.db
python listen.py baseline --duration 600                       # stop after 10 min
python listen.py playback --manifest session.json --db events.db
```

There is no build step and no test suite yet. Syntax-check all modules with:

```bash
python3 -c "import ast, sys; [ast.parse(open(f).read()) or print('OK', f) for f in ['config.py','detector.py','storage.py','session.py','audiowriter.py','videowriter.py','listen.py','evaluate.py']]"
```

## Architecture

The pipeline per 42.7 ms audio chunk (8192 samples @ 192 kHz):

```
sounddevice callback (audio thread)
  └─ compute_power_spectrum(chunk)     # Hann-windowed rfft → dBFS array
       └─ BandDetector.process(ps, t)  # one instance per monitored band
            └─ queue.SimpleQueue       # events handed off to main thread
                 └─ EventLog.log_event → SQLite
```

**`config.py`** is the single source of truth for all constants — band Hz ranges, `SAMPLE_RATE`, `CHUNK_SAMPLES`, and detection thresholds. Change values here; nothing is hard-coded elsewhere.

**`detector.py`** — `BandDetector` maintains a rolling 50-chunk noise floor (median of per-chunk band medians) and an onset/offset state machine. A call opens when peak band power exceeds `DETECTION_THRESHOLD_DB` above the floor, **and** passes the call-shape gate, and closes after `CALL_END_SILENCE_MS` of silence. Call `flush()` before closing the stream to capture any in-progress call. `compute_power_spectrum` is a module-level function used by `listen.py`; it is not a method of `BandDetector`.

**Call-shape gate** (`CALL_SHAPE_GATE_ENABLED` in `config.py`) — broadband transients (e.g. music-playback-chain distortion, percussive hash) can cross `DETECTION_THRESHOLD_DB` without being a narrowband FM call. `band_shape_features()` computes, per chunk, an occupied-band fraction and the widest contiguous run of bins within `BANDWIDTH_DROPOFF_DB` of *the chunk's own peak power in that band* (a standard "-X dB bandwidth" measure), plus peak-to-median concentration in dB. `passes_call_shape_gate()` rejects onset when the occupied run spans more than `MAX_CALL_BAND_FRACTION` of the band or concentration falls below `MIN_PEAK_CONCENTRATION_DB`. The occupied-bin threshold is deliberately peak-relative, not floor-relative: an earlier floor-relative "shoulder" design saturated to ~100% occupancy on real 22 kHz alarm calls (general chunk-wide energy elevation above the floor, not just the call, was miscounted as occupied), which cut recall to ~50% at defaults — measured with `evaluate.py` against the Olszynski & Polowy alarm playback WAVs and the tickling recording. Current defaults preserve ~93-100% recall on those two real positive corpora. **Now measured against a real music negative** (41.5 min of an ordinary FLAC album played through laptop speakers — not USV-enriched, and not played through the deployment's ultrasonic Vifa-class speakers — captured live through the SO.104 via `listen.py record`): rejection at the current defaults (`MAX_CALL_BAND_FRACTION=0.9`, `MIN_PEAK_CONCENTRATION_DB=6.0`) is **0.0%** — every false-positive chunk the music produced also passed the shape gate, while real-call recall stayed ~97%. `evaluate.py --sweep` shows `max_run_fraction` has no discriminating effect on these false positives anywhere in 0.2-0.9 (both real-call and false-positive candidate chunks sit well under even the tightest value); `min_peak_concentration_db` is the only lever with any effect, but the trade-off is bad throughout the tested range (12.0 dB → ~4% rejected at ~50-53% recall; 18.0 dB → ~55% rejected at ~41-43% recall). Laptop speakers cannot radiate meaningful acoustic energy in 18-80 kHz, so these false positives are not acoustic instrumental ultrasound — near-uniform counts across every monitored band (see Frequency bands section below) point to a playback+capture-chain artifact (harmonic/intermodulation distortion and/or electrical/USB coupling between the laptop and the SO.104) rather than content encoded in the source FLAC. Conclusion: this gate distinguishes narrowband calls from broadband/percussive noise (its original design target) but not from this chain artifact, whose false-positive chunks are themselves narrowband and concentrated — a real limitation, not just an untested caveat, though whether genuine ultrasonic-radiating music (played through the deployment's actual ultrasonic speakers) would fare any differently remains untested. `BandDetector` also exposes `last_peak_excess` / `last_shape_features` / `last_is_active` / `last_peak_freq` / `last_peak_power_db` as read-only diagnostics for `evaluate.py`; the live pipeline doesn't use them.

**Sub-harmonic gate** (`SUBHARMONIC_GATE_ENABLED` in `config.py`, default **off**) — a third candidate discriminator tried against the same music negative, since the two features above measured 0% rejection on it. `harmonic_features()` searches the *full* spectrum (not just the band slice) for energy at `f_peak/2` and `f_peak/3`: rat USVs are near-pure whistle tones with no sub-harmonic content, while the music-playback false positives almost always carry real energy at those sub-multiples — consistent with harmonic/intermodulation distortion products of the loud audible-band music generated downstream in the playback+capture chain, not with content encoded in the source FLAC. `subharmonic_ratio_db` (strongest sub-harmonic power minus the chunk's own peak power) showed a large **per-chunk** median gap for `alarm` (call -46.8 dB vs. false-positive +4.0 dB) and `social_low` (call -27.7 dB vs. false-positive -0.3 dB) — but not `social_core` (call p90 was +42.6 dB; complex 50 kHz modulated/trill calls often carry real sub-multiple energy), so the gate is scoped to `alarm`/`social_low` via `SUBHARMONIC_GATE_BANDS`. The per-chunk percentiles were misleading, though: a confirmatory `evaluate.py` run at `MAX_SUBHARMONIC_RATIO_DB=-1.5` (the ~90th-percentile boundary of real-call chunks) measured **event-level** recall of only ~78% (396/508) against ~58% rejection of the false positives (2080/4938, vs. 0% today) — real calls are frequently only 1-3 chunks long, so losing even one chunk to a per-chunk hard cutoff can kill or truncate the whole call below `MIN_CALL_DURATION_MS`. That's the best rejection lever measured so far (better than `MIN_PEAK_CONCENTRATION_DB`'s 55%-reject/41-43%-recall option), but still short of the ~90% recall bar this project requires before shipping a gate on by default, so it ships present-but-disabled. A per-*call* version (requiring several consecutive failing chunks, not one) is the likely next step if revisited. A fourth candidate, per-call FM (peak-frequency) modulation depth/range, was also measured and rejected outright — pos/neg distributions overlapped too heavily to discriminate at all (e.g. alarm: call median 5.88 kHz range vs. false-positive median 4.45 kHz; social_low: call median 12.0 kHz vs. false-positive median 9.94 kHz) — no code shipped for it beyond the harness measurement in `evaluate.py`. A fifth candidate, **temporal continuity** — chunk-to-chunk peak-frequency *step* `|f_{i+1}-f_i|` (connectedness of the peak contour, distinct from FM total range), the discriminator the shape-gate note above speculated would work — was measured against the same corpora using the detector's active-chunk-only `peak_freq_trace` (excludes trailing offset-silence chunks whose "peak" is noise argmax) and **rejected, because the separation runs backwards**: `median_step_hz(>=3)` positive p90 sits 3-4x *above* negative p50 in every band (alarm pos 1640/3830 vs. neg 1050/3150; social_low pos 1980/7460 vs. neg 2330/6580; social_core pos 3890/11000 vs. neg 2840/8650 Hz, p50/p90). Real social calls sweep/trill (large steps) while the surviving false positives are sustained *held* tones (flat → small steps) — consistent with a steady chain-distortion artifact riding on sustained musical passages — so continuity is more call-like for the artifact than for the calls. It also can't apply to most alarm calls (only 107/263 confirmed alarm events have ≥3 chunks — 59% are 1-2 chunks, the same short-call fragility that sank the sub-harmonic gate). **Net result of Part A: no single-source spectral or temporal feature tried (occupied-run, concentration, sub-harmonic, FM-range, continuity) separates a rat whistle from this laptop-playback chain artifact — the remaining path is context (clean-by-design alarm band during playback, plus Phase 3 spectrogram masking), not classification. Whether genuine ultrasonic-radiating music, played through the deployment's actual ultrasonic speakers, would behave the same way is untested — `music_neg` (ordinary FLAC through laptop speakers, no USV-band source content) cannot answer that.** The diagnostic `peak_freq_trace` field on `DetectionEvent` (active-chunk peak-freq contour) feeds this measurement in `evaluate.py` and is otherwise runtime-inert, like the other `last_*` diagnostics.

**`session.py`** — reads the Squeakorithm manifest JSON and answers `current_track(epoch_time) → (TrackInfo, elapsed_s)`. Used only in playback mode to attach track-relative timestamps to events.

**`storage.py`** — thread-safe `EventLog` wraps SQLite with a mutex. The consumer thread (not the audio callback) calls `log_event`; the audio callback only enqueues `DetectionEvent` objects. Events include `audio_path TEXT`, `video_path TEXT`, and `valence TEXT` columns. On init, the schema runs a migration: any of these columns absent in an existing DB is added via `ALTER TABLE`.

**`audiowriter.py`** — `AudioRingBuffer` retains the last `AUDIO_BUFFER_SECONDS` (30 s) of raw mic chunks tagged by index; `AudioClipWriter` writes lossless FLAC clips to `audio/{band}/{datetime}.flac`. Both classes are guarded by `threading.Lock`, mirroring `EventLog`'s mutex pattern.

**`videowriter.py`** — optional video capture (enabled with `--video`). `VideoRingBuffer` stores the last `VIDEO_BUFFER_SECONDS` (30 s) of camera frames as JPEG bytes (~135 MB vs ~2.4 GB raw) tagged by Unix timestamp. `VideoCaptureThread` is a daemon thread that opens a UVC camera via OpenCV (`cv2.VideoCapture`), runs at ~30 fps, and feeds the ring buffer; if the camera fails to open it prints a warning and disables video without crashing the audio pipeline. `VideoClipWriter.write_clip` extracts a ±5 s window around each call and writes an mp4. Bout de-duplication skips overlapping windows so a burst of calls produces one clip, not dozens. Designed for IR-modified USB cameras under infrared illumination — camera-agnostic via OpenCV index or path.

**`listen.py`** — audio callback stays thin: ring-buffer append + FFT + enqueue only. A daemon consumer thread drains the queue and handles all I/O (print + SQLite + FLAC write + mp4 write). Timestamps use PortAudio's ADC clock calibrated on first callback invocation (`inputBufferAdcTime`) rather than `time.time()` per chunk. Pass `--no-audio` to disable audio clips; `--audio-dir` overrides the default `audio/` directory. Pass `--video` to enable video clips (off by default); `--video-dir`, `--video-continuous`, and `--camera` control the camera index/path, output directory, and whether a continuous full-session mp4 is also written. Each event is tagged with a `valence` label via `detector.classify_valence()` — a minimal band/peak-frequency heuristic (22 kHz alarm and 44 kHz → `negative`, 50–70 kHz → `positive`, else `unknown`), unvalidated groundwork rather than a trusted classifier. Checked against the real Olszynski/Polowy playback WAVs via `evaluate.py --valence-check`: the 22 kHz and 44 kHz files classify correctly (`negative`), confirming the heuristic's logic is sound where it's reachable. `BASELINE_BANDS` now includes `social_core`, so real 50 kHz (positive-register) calls correctly produce `positive` in baseline mode (confirmed: all four 50-kHz playback WAVs now show real `positive` counts). `PLAYBACK_BANDS` still omits `social_core` (see Frequency bands section below), so during Squeakorithm playback the same leak into `social_low` and mislabeling as `negative`/`unknown` remains until Phase 3 masking. The `record` subcommand writes raw mic input straight to a WAV with no detection/gate applied — used to capture negative corpora (e.g. music played back with no rat present) for `evaluate.py`. Note: its `while frames_written < total_frames` loop has no stall/idle watchdog — if the input stream stops delivering callbacks partway through, the process hangs indefinitely rather than exiting, and must be killed manually (the WAV file up to that point remains valid).

**`evaluate.py`** — standalone offline harness that replays labeled WAV corpora through the real `compute_power_spectrum` + `BandDetector` pipeline to measure and tune the call-shape gate. Resamples every input file to `config.SAMPLE_RATE` via `scipy.signal.resample_poly` (eval-only dependency, not in `requirements.txt`'s runtime set; also skips macOS AppleDouble `._*` sidecar files when scanning a directory). `--gate on|off` toggles `CALL_SHAPE_GATE_ENABLED` for a single-run report of per-file/per-band detection counts; `--sweep` grid-searches `MAX_CALL_BAND_FRACTION` × `MIN_PEAK_CONCENTRATION_DB` and reports positive-recall-kept vs. negative-rejected fractions relative to a gate-off baseline, plus prints raw shape-feature percentiles (pos vs. neg, now including `subharmonic_ratio_db`) so thresholds are chosen from data. `replay()` also returns `fm_records` — one entry per *confirmed* event with peak-frequency range/std across that event's chunks, printed via `print_fm_summary()` for the (rejected) FM-modulation-depth candidate feature. `--valence-check` replays each input file and tallies `detector.classify_valence()` label counts per file instead of a detection-count report. **`--sweep` replays the full corpus once per grid combination** (31 full passes at the default 6×5 grid) — on more than a few minutes of audio this gets slow (a full 41.5-minute corpus pass takes roughly 8-9x its own duration to process, so a 31-combo sweep over it is multi-hour); trim inputs to a representative clip and/or pass `--sweep-fractions`/`--sweep-concentrations` with fewer values for an interactive-scale run. `--artifact-check` (`artifact_features()`/`print_artifact_summary()`) is a lighter, detector-independent diagnostic for characterizing *what* a negative source's in-band energy is: per band it reports mean peak power (band-uniformity), the Pearson correlation between that band's per-chunk peak power and a reference audible-band (1-15 kHz, inside a laptop speaker's passband) peak power, and the fraction of the band's loudest 10% of chunks whose peak frequency sits within 2% of an integer multiple of the chunk's audible-band peak frequency (harmonic-comb match). Run against the real `music_neg.wav` (41.5 min, ordinary FLAC through laptop speakers): audible-ultrasonic correlation was 0.61-0.68 across alarm/social_low/social_core, and the harmonic-comb match was 40.7% (alarm), 64.4% (social_low), 79.4% (social_core) — a meaningfully high fraction of the loudest candidate chunks in every band sit at an integer multiple of the audible-band fundamental. This corroborates the chain-artifact framing above with a direct spectral signature, rather than only the indirect argument from near-uniform event counts and physical implausibility of laptop-speaker ultrasound.

## Frequency bands and overlap

Squeakorithm enforces a **33 kHz hard floor** on all USV generators, so the **alarm band (18–26 kHz) is always clean** — no music energy, no masking needed. `social_low` (30–45 kHz) has a clean wedge at 30–33 kHz but overlaps music above 33 kHz.

`BASELINE_BANDS` in `listen.py` is `["alarm", "social_low", "social_core"]`; `PLAYBACK_BANDS` remains `["alarm", "social_low"]`. `social_core` (45–70 kHz) is the only band where `classify_valence()`'s positive/appetitive branch can fire, so monitoring it is what makes positive valence reachable at all — safe in baseline mode (no Squeakorithm playback ⇒ no 33 kHz-floor music energy to contend with). It stays out of `PLAYBACK_BANDS` (along with `social_high`, 70–80 kHz) until Phase 3 spectrogram masking, since both overlap the 33–80 kHz music band during actual playback. Measured exposure: over the 41.5-min `music_neg` corpus, `social_core` produced 2468 false-positive events at gate-off (comparable to alarm's 2452 and social_low's 2486) — so a baseline room is only as clean for `social_core` as it is for the other bands; the same call-shape/sub-harmonic gates apply to it as to any other band. One known limitation introduced by adding `social_core`: it overlaps `social_low`'s upper edge and `pup`, so a single 50 kHz call can now fire in both `social_low` and `social_core` — the `social_core` event gets the correct `positive` label, but the co-fired `social_low` event may still mislabel. No cross-band dedup was attempted (Phase 3 territory).
